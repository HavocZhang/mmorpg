2. 编译构建

开发编译（快速迭代）

# 开发模式编译（opt-level=1, debug=true, incremental=true） $ cargo build

生产编译（极致优化）

# Release 模式（opt-level=3, LTO=true, codegen-units=1, panic=abort） $ cargo build --release # 或使用一键脚本（含 clippy 零警告检查） $ bash build.sh

仅检查编译错误（不生成二进制）

# 快速检查，不含测试目标 $ cargo check # 检查所有目标（含测试、benchmark） $ cargo check --all-targets # Windows 下建议限制并行编译数（内存不足时） $ CARGO_BUILD_JOBS=2 cargo check --all-targets

5. 测试体系（TDD + BDD）

项目采用 四层测试金字塔，覆盖从单元到行为验收的完整链路：

📊 测试架构总览
层级目录内容运行命令
TDD 单元 tests/tdd_unit/ 协议编解码、CRC32、Snowflake ID、错误分类 cargo test --test protocol_tests
TDD 并发 tests/tdd_concurrent/ DashMap 并发读写、RateLimiter 多线程、Snowflake 唯一性 cargo test --test concurrency_tests
TDD 异常 tests/tdd_exception/ 空包/半包/超大包/篡改包/AES 解密失败 cargo test --test exception_tests
TDD 模糊 tests/tdd_fuzz/ 100 组随机数据喂入解码器、边界值测试 cargo test --test fuzz_tests
BDD 场景 tests/bdd_feature/ 7 个 .feature 文件（连接/会话/协议/消息/安全/集群/停机） 需安装 cucumber 依赖后 cargo cucumber
Benchmark benches/gate_bench.rs AES 加解密、CRC32、编解码、合并、限流性能基准 cargo bench

5.1 运行全部测试

# 运行所有 TDD 测试（单元 + 并发 + 异常 + 模糊） $ cargo test --all --no-fail-fast # 或使用一键脚本 $ bash test.sh # Windows 内存不足时限制并行编译 $ CARGO_BUILD_JOBS=2 cargo test --all --no-fail-fast

5.2 运行单个测试文件

# 只跑单元测试 $ cargo test --test protocol_tests # 只跑并发测试 $ cargo test --test concurrency_tests # 只跑异常测试 $ cargo test --test exception_tests # 只跑模糊测试 $ cargo test --test fuzz_tests

5.3 运行库内嵌测试

# 运行 src 内 #[cfg(test)] 模块（如 config::app_config 的测试） $ cargo test --lib

5.4 运行指定测试函数

# 按函数名过滤 $ cargo test --test protocol_tests -- test_encode_decode_roundtrip $ cargo test --test concurrency_tests -- test_concurrent_snowflake # 显示 println! 输出 $ cargo test -- --nocapture

5.5 各测试文件详情

单元测试 protocol_tests.rs — 8 个测试用例
  • test_encode_decode_roundtrip — 编码后解码，验证数据一致性
  • test_header_magic — 验证包头 Magic 字段（0x4D4D）
  • test_max_body_size_constant — 验证常量 MAX_BODY_SIZE=8192, HEADER_SIZE=16
  • test_multiple_packets_decode — 10 个包拼接后一次性解码（粘包处理）
  • test_crc32_verification — CRC32 校验通过 / 篡改后校验失败
  • test_snowflake_id_generation — 连续生成 ID 不重复
  • test_error_types — GateError 安全性分类与可恢复性判断
并发测试 concurrency_tests.rs — 3 个测试用例
  • test_concurrent_ip_blacklist — 8 线程并发写入 800 条 IP 黑名单，无竞争
  • test_concurrent_rate_limiter — 8 线程并发限流检查，无 panic
  • test_concurrent_snowflake — 8 线程 × 1000 次 ID 生成，8000 个 ID 全局唯一
异常测试 exception_tests.rs — 8 个测试用例
  • test_decode_empty_buffer — 空缓冲区返回 None
  • test_decode_partial_header — 半包头（8 字节）返回 None
  • test_decode_invalid_magic — 错误 Magic 返回 Err
  • test_decode_oversized_packet — 超过 8192 字节返回 PacketTooLarge
  • test_decode_crc_mismatch — 篡改包体后 CRC 校验失败
  • test_aes_decrypt_tampered — 篡改密文后 AES 解密失败
  • test_aes_decrypt_short_data — 过短密文解密失败
  • test_invalid_aes_key — 非法密钥格式报错
模糊测试 fuzz_tests.rs — 5 个测试用例
  • test_fuzz_random_data — 100 组随机长度随机数据，不 panic
  • test_fuzz_empty_packets — 空数据 / 单字节数据
  • test_fuzz_partial_headers — 1~15 字节不完整包头
  • test_fuzz_valid_magic_bad_rest — 有效 Magic + 随机剩余字节
  • test_fuzz_max_size_boundary — 8192 边界值 + 8193 超限值

9. 常见问题

Q: 编译报内存不足 / ICE（内部编译器错误）？
A: 限制并行编译数：CARGO_BUILD_JOBS=2 cargo check。项目已固定 Rust 1.95.0， 避免新版本 ICE 问题。
Q: 测试编译很慢？
A: 首次编译需要编译所有依赖 + 测试代码，单线程约需 10+ 分钟。建议用 CARGO_BUILD_JOBS=2 cargo test，后续增量编译会快很多。
Q: 如何添加新的测试用例？
A: 单元测试直接在 src/ 模块内用 #[cfg(test)] mod tests； 集成测试在 tests/ 对应目录新建 .rs 文件； 行为测试在 tests/bdd_feature/ 新建 .feature 文件。
Q: 如何查看 Prometheus 监控指标？
A: 启动网关后访问 http://localhost:9090/metrics， 可对接 Grafana 进行可视化。
